Accessibilité et routage côté client

Les différents frameworks et plugins JavaScript mettent en œuvre le routage côté client de manière différente, mais ils ont tous en commun de désactiver le mécanisme de navigation de site par défaut du navigateur et d'injecter du contenu dans le DOM. La gestion manuelle du routage peut entraîner des changements inattendus ou indiscernables. Pour les utilisateurs qui dépendent principalement de l'accessibilité au clavier, le focus peut ne pas se trouver là où ils l'attendent. Il peut même être complètement perdu, ce qui peut rendre la navigation beaucoup plus difficile. Les utilisateurs de lecteurs d'écran peuvent ne pas remarquer que le contenu d'une page a changé en raison de l'absence de retour d'information.

La réponse la plus évidente à ce problème est d'éviter de le provoquer en premier lieu et de se fier au comportement de navigation par défaut du navigateur. Mais selon les exigences ou la pile technologique, ce n'est pas toujours possible ou souhaitable.

Commençons par un exemple simple. Vous désactivez le comportement par défaut du navigateur et, au lieu de recharger la page, vous ajoutez le contenu d'un modèle à l'élément principal (voir les exemples 1 et 2).

Exemple 1 – Une navigation, un élément principal vide et le contenu des deux pages liées stocké dans une balise template

<header>
  <nav>
    <ul>
      <li><a href="#page-home">accueil</a></li>
      <li><a href="#page-about">à propos de</a></li>
    </ul>
  </nav>
</header>

<main>
</main>

<template id="page-home">
  <h1 id="heading">Accueil</h1>
</template>

<template id="page-about">
  <h1 id="heading">À propos de nous</h1>
</template>

Exemple 2 – Solution incomplète : Un événement de clic empêche le comportement par défaut des liens et clone le contenu du modèle avec un id correspondant dans l'élément principal

const main = document.querySelector("main");
const nav = document.querySelector("nav");

nav.addEventListener("click", (e) => {
    if (e.target.nodeName === "A") {
        e.preventDefault();
        const id = e.target.getAttribute("href");
        const page = document.querySelector(id);

        main.innerHTML = "";
        main.appendChild(page.content.cloneNode(true));
    }
});

Cela fonctionne bien pour les utilisateurs de souris et les utilisateurs tactiles, mais les utilisateurs de lecteurs d'écran n'ont pas de retour d'information lorsque le contenu du DOM change, et l'attention est toujours portée sur le lien.

Gestion de l'attention

Une solution à ce problème consiste à déplacer l'attention du lien vers un élément sémantique étiqueté, qui pourrait être l'élément principal, comme dans les exemples 3 et 4.

Exemple 3 –  Une solution simple de routage côté client avec gestion de la focalisation

<header>
  <nav>
    <ul>
      <li><a href="#page-home">accueil</a></li>
      <li><a href="#page-about">à propos de</a></li>
    </ul>
  </nav>
</header>

<main  tabindex="-1" aria-labelledby="heading">
</main>

<template id="page-home">
  <h1 id="heading">Accueil</h1>
</template>

<template id="page-about">
  <h1 id="heading">À propos de nous</h1>
</template>

Note : L'élément main est focalisable et étiqueté par ses enfants

Exemple 4 – Le script met l'accent sur l'élément principal après avoir ajouté le contenu du modèle.

const main = document.querySelector("main");
const nav = document.querySelector("nav");

nav.addEventListener("click", (e) => {
    if (e.target.nodeName === "A") {
        e.preventDefault();
        const id = e.target.getAttribute("href");
        const page = document.querySelector(id);

        main.innerHTML = "";
        main.appendChild(page.content.cloneNode(true));
                main.focus();
    }
});

Note : main.focus() est ajouté

Lorsqu'un utilisateur clique sur un lien, la mise au point se déplace vers la zone de contenu principale de la page. Un effet secondaire bienvenu de la mise au point d'un élément avec JavaScript est que les lecteurs d'écran annoncent automatiquement son rôle et son nom accessible. Selon le lecteur d'écran et le navigateur, cet exemple annonce quelque chose comme "Principal, Accueil" ou "Principal, À propos".

Régions dynamiques

Une autre solution à ce problème consiste à signaler les changements de page à l'aide d'une région vivante. Les régions en direct sont des régions perceptibles qui annoncent aux lecteurs d'écran les changements apportés à leur contenu à la suite d'un événement externe, alors que l'attention de l'utilisateur peut être ailleurs. Elles peuvent être utiles pour communiquer les changements de DOM, les résultats des interactions avec l'utilisateur, l'état d'attente, la progression du processus, les erreurs ou les changements de page. Parmi les exemples, on peut citer les journaux de discussion, la validation des formulaires ou le routage côté client.

Chaque fois que vous modifiez le contenu de l'élément avec role="status" (cet attribut et cette valeur le transforment en région vivante), un lecteur d'écran l'annonce automatiquement. Les exemples 5 et 6 montrent une solution simple de routage côté client utilisant une région vivante.

Exemple 5 –  Une solution simple de routage côté client avec une région dynamique

<header>
  <nav>
    <ul>
      <li><a href="#page-home">accueil</a></li>
      <li><a href="#page-about">à propos de</a></li>
    </ul>
  </nav>
</header>

<main></main>
<div role="status" class="visually-hidden"></div>

<template id="page-home">
  <h1 id="heading">Accueil</h1>
</template>

<template id="page-about">
  <h1 id="heading">À propos de nous</h1>
</template>

.visually-hidden { 2
  clip-path: inset(50%);
  height: 1px;
  overflow: hidden;
  position: absolute;
  white-space: nowrap;
  width: 1px;
}

Note : L'élément <div> annonce les changements sur la page (la région en direct). Le code css permet de déplacer visuellement un élément hors de la fenêtre.

Exemple 6 –  Le script modifie le contenu de la région en direct après avoir ajouté le contenu du modèle

const main = document.querySelector("main");
const nav = document.querySelector("nav");
const region = document.querySelector('[role="status"]');

nav.addEventListener("click", (e) => {
    if (e.target.nodeName === "A") {
        e.preventDefault();
        const link = e.target;
        const id = link.getAttribute("href");
        const page = document.querySelector(id);

        main.innerHTML = "";
        main.appendChild(page.content.cloneNode(true));
        region.textContent = `${link.textContent}`; 
    }
});

Note : region.textContent modifie le contenu de la région en direct chaque fois que l'utilisateur clique sur un lien.

L'accent est toujours mis sur le lien, alors vous pouvez envisager de combiner les deux solutions si nécessaire, mais n'oubliez pas que cela pourrait entraîner une redondance : un lecteur d'écran pourrait transmettre le contenu dans la région en direct et le retour d'information en mettant l'accent sur un élément.

Ce ne sont que des exemples. Il existe de nombreuses façons différentes de mettre en œuvre et de combiner les deux solutions, et vous ne ppouvez déterminer ce qui fonctionne le mieux qu'en effectuant des tests avec les utilisateurs.

La plupart des éléments en HTML sont accessibles par défaut, ce qui signifie qu'ils fournissent d'emblée les informations sémantiques et les fonctionnalités nécessaires à la plupart des utilisateurs. À part quelques exceptions, se fier au comportement par défaut des éléments HTML est une valeur sûre. Lorsque vous désactivez ces valeurs par défaut, vous devez vous assurer que votre solution personnalisée offre une expérience utilisateur similaire ou meilleure.

Prenons l'exemple de la navigation sur les pages. Lorsqu'un utilisateur clique sur un lien, le navigateur charge une nouvelle page, modifie le document affiché dans la fenêtre de visualisation et met l'accent sur le document. S'il utilise un lecteur d'écran, il annonce également le titre de la page (<title>), qui doit être unique sur chaque page. Quelle que soit la façon dont ils accèdent au site Web, ils savent généralement toujours ce qui s'est passé, qu'il s'est passé quelque chose et où ils se trouvent. Aucune de ces choses ne se produit lorsque vous empêchez le comportement de clic par défaut d'un lien. Vous devez soit les recréer manuellement, soit proposer une alternative similaire ou meilleure.

Gestion de l'attention

Le comportement par défaut du navigateur , qui consiste à se concentrer sur le document lors du chargement d'une nouvelle page. Est-il le meilleur pour l'expérience de l'utilisateur ? C'est discutable. En comparant différentes solutions, j'ai constaté que cela dépendait de la méthode de navigation de l'utilisateur. Toujours est-il que dans une solution de rendu personnalisée côté client, vous devez gérer le focus d'une manière ou d'une autre et le déplacer vers un élément focalisable et sémantique. Pour ce faire, placez l'attribut tabindex sur l'élément et fixe la valeur à -1, ce qui garantit que vous pouvez le focaliser via la méthode focus() de JavaScript, mais pas avec la touche Tab.

L'élément focalisable ne doit pas être un élément générique, comme div, car les lecteurs d'écran les traitent différemment et vous ne pouvez pas les nommer. Voir l'exemple 7.

Exemple 7 –  Mauvaise pratique : Ne mets pas l'accent sur un espace vide. div

<div tabindex="-1"></div>

L'élément focalisable peut être l'élément <main>, comme le montre l'exemple 3, un <section>, ou un <hn>. Il a idéalement un nom accessible, provenant soit de son contenu, soit de aria-label, comme le montre l'exemple 8. Vous ppouvez aussi utiliser aria-labelledby pour créer une référence à un élément existant. Dans le cas de l'exemple 3, il s'agirait de l'en-tête de chaque page.

Exemple 8 –  Une section étiquetée focalisable

<section tabindex="-1" aria-label="Produits"></section>

À propos des régions dynamiques

Vous pouvez créer et configurer des régions dynamiques de différentes manières. Les attributs les plus pertinents sont aria-live, aria-atomic, et aria-relevant.

Vous pouvez transformer un élément en une région en direct en définissant l'attribut aria-live sur assertive ou polite. Les mises à jour assertives ont la plus haute priorité et doivent être présentées immédiatement, en interrompant les annonces en cours si nécessaire. Les mises à jour polite sont moins agressives et attendent la prochaine occasion possible, comme la fin d'une phrase ou les pauses de frappe.

Par défaut, seuls les nœuds DOM modifiés seront présentés car aria-atomic est false. Dans l'exemple 9, vous modifiez la valeur d'un élément <span> à l'intérieur d'une région dynamique au moment du clic. Le lecteur d'écran n'annonce que le contenu du nœud DOM modifié, par exemple "Olivia", et non la phrase entière. Il ne le fait également que si le texte change. Il n'y a pas d'annonce si la valeur actuelle est "Philippa" et que la nouvelle valeur qui la remplace est la même.

Exemple 9 –  Mise à jour d'un nœud imbriqué dans une région en direct

<button type="button">Changez le nom</button>

<div aria-live="polite">
  Mon nom est <span>Claudia</span>
</div>

<script>
const button = document.querySelector("button");

// liste de noms
const names = [
    "Maurice",
    "Valentine",
    "Johanna",
    "Magalie",
    "Victoria",
    "Philippe",
    "Olivia",
];
const name = document.querySelector("span");

button.addEventListener("pointerdown", e => {
   const random = Math.floor(Math.random() * names.length);
   name.textContent = names[random]; // remplit l'élément span avec un nom choisi au hasard. 
});
</script>

Si vous ajoutez l'attribut aria-atomic et que vous le définissez sur true, vous rendez la région dynamique atomique, ce qui signifie que les lecteurs d'écran présentent la région entière, par exemple, "Mon nom est Maurice", dans l'exemple 10.

Exemple 10 –  Mise à jour d'un nœud imbriqué dans une région en direct

<button type="button">Changez le nom</button>

<div aria-live="polite" aria-atomic="true">
  Mon nom est <span>Claudia</span>
</div>

Il existe également des rôles de région dynamique dédiés, par exemple status et alert, et un élément de région dynamig, <output>. L'attribut role="status" est un raccourci pour aria-live="polite" et aria-atomic="true" (voir l'exemple 11) et role="alert" est un raccourci pour aria-live="assertive" et aria-atomic="true".

Exemple 11 –  Une région dynamique

<button type="button">Changez le nom</button>

<div  role="status">
  Mon nom est <span>Claudia</span>
</div>

L'élément de sortie (exemple 12) possède un aria-live="polite" et un aria-atomic="true" implicites.

Exemple 12 –  Utilisation de output comme région dynamique

<button type="button">Changez le nom</button>

<output>
  Mon nom est <span>Claudia</span>
</output>

Le titre de la page

Il ne suffit pas d'annoncer un changement de page : vous devez aussi changer le titre du document <title>. Les utilisateurs de lecteurs d'écran utilisent des raccourcis pour lire le titre de la page, ce qui peut être utile pour s'orienter. (Vous trouverez plus d'informations sur les titres de page dans cet article : Décrire le document).

La navigation de page côté client est un sujet complexe. Les solutions proposées dans cet article ne sont pas des snippets prêts à l'emploi que vous pouvez mettre en œuvre sur vos sites Web. Elles illustrent simplement le problème sous-jacent et sa complexité. La façon dont vous le résolvez dépend de de votre framework et, surtout, de ce qui fonctionne le mieux pour tes utilisateurs. Lorsque vous utilisez un framework ou que vous choisissez un routeur pour votre application, assurez-vous qu'ils le font de manière accessible ou qu'ils fournissent au moins des instructions sur la manière de le faire.
