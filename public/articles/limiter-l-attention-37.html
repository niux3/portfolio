Limiter l'attention

Si vous déplacez le focus, vous devez vous assurer que le reste de la page est inactif afin que l'utilisateur ne puisse interagir qu'avec le contenu actuellement pertinent. Si vous ne le faites pas, l'utilisateur peut accidentellement sortir de cette zone, se perdre et avoir du mal à retrouver son chemin, comme le montrent les exemples 1, 2 et 3. La clef est de contenir ou de pièger l'attention dans la zone actuellement pertinente de la page et rendre le reste inaccessible.

Exemple 1 – Focus contenir dans une boîte de dialogue modale personnalisée

<button type="button" class="open" aria-haspopup="dialog">Se connecter</button>
<div role="dialog" aria-modal="true" hidden aria-labelledby="heading">
  <div>
    <button type="button" class="close">Fermer</button>
    <h1 id="heading" tabindex="-1">Se connecter</h1>
    <a href="#">Un lien</a>
  </div>
</div>
<script>
  const dialogOpen = document.querySelector(".open");
  const dialogClose = document.querySelector(".close");
  const dialog = document.querySelector('[role="dialog"]');
  const heading = dialog.querySelectorAll('[tabindex="-1"]')[0];
  // Trouve tous les éléments sur lesquels on peut mettre l'accent dans le dialogue.
  const focusableElements = dialog.querySelectorAll("button, a");
  let trigger;
  dialogOpen.addEventListener("click", open);
  dialogClose.addEventListener("click", close);
  dialog.addEventListener("keydown", (e) => {
    // Ne rien faire si l'utilisateur n'appuie pas sur la touche Tab.
    if (e.code !== "Tab") return;
    // C'est le premier élément sur lequel on peut mettre l'accent dans le dialogue.
    const first = focusableElements[0];
    // C'est le dernier élément sur lequel on peut mettre l'accent dans la boîte de dialogue.
    const last = focusableElements[focusableElements.length - 1];
    // Il s'agit de l'élément sur lequel on se concentre actuellement.
    const active = document.activeElement;
    // Lorsque l'utilisateur appuie sur la touche Shift
    if (e.shiftKey) {
      // et l'élément sur lequel on se concentre actuellement est le premier élément
      if (first === active) {
        e.preventDefault();
        // empêcher le comportement par défaut et déplacer le focus sur le dernier élément sur lequel le focus est possible.
        last.focus();
        8
      }
    } else if (last === active) {
      // Si l'utilisateur n'appuie pas sur Shift et que le dernier élément est l'élément sur lequel on se concentre actuellement, le focus se déplace sur le premier élément.
      e.preventDefault();
      first.focus();
    }
  });

  function open() {
    trigger = document.activeElement;
    dialogToggle();
  }

  function close() {
    dialogToggle();
  }

  function dialogToggle() {
    const isOpen = !dialog.hasAttribute("hidden");
    if (!isOpen) {
      dialog.removeAttribute("hidden");
      heading.focus();
    } else {
      dialog.setAttribute("hidden", "hidden");
      trigger.focus();
    }
  }
</script>

Exemple 2 – Focus contenu dans un élément de dialogue personnalisé

<div class="page-wrapper">
    <!-- Tout le contenu de la page principale est contenu dans un seul élément. -->
    <header>
    <a href="#">Accueil</a>
  </header>

  <main>
    <button class="open" aria-haspopup="dialog">Se connecter</button>
  </main>
</div>
<div role="dialog" aria-modal="true" hidden aria-labelledby="heading">
  <div>
    <button type="button" class="close">Fermer</button>
    <h1 id="heading" tabindex="-1">Se connecter</h1>
    <a href="#">Un lien</a>
  </div>
</div>
<script>
const pageWrapper = document.querySelector('.page-wrapper');
const dialogOpen = document.querySelector(".open");
const dialogClose = document.querySelector(".close");
const dialog = document.querySelector('[role="dialog"]');
const heading = dialog.querySelectorAll('[tabindex="-1"]')[0];
let trigger;

dialogOpen.addEventListener("click", open);
dialogClose.addEventListener("click", close);

function open() {
  trigger = document.activeElement;
    // L'enveloppe du contenu principal devient inerte lorsque la boîte de dialogue s'ouvre.
  pageWrapper.setAttribute('inert', 'inert') 
  dialogToggle();
}

function close() {
  // Le statut inerte est supprimé lorsque la boîte de dialogue se ferme.
  pageWrapper.removeAttribute('inert') 
    dialogToggle();
}

function dialogToggle() {
  const isOpen = !dialog.hasAttribute("hidden");

  if (!isOpen) {
    dialog.removeAttribute("hidden");
    heading.focus();
  } else {
    dialog.setAttribute("hidden", "hidden");
    trigger.focus();
  }
}
</script>

Exemple 3 – Focus contenu nativement dans un élément de dialogue

<button type="button" class="open">Se connecter</button>

<dialog tabindex="-1" aria-labelledby="heading">
  <form method="dialog">
    <button type="button">Fermer</button>
  </form>

  <h1 id="heading">Se connecter</h1>
</dialog>

<script>
const button = document.querySelector("button");
const dialog = button.nextElementSibling;

button.addEventListener("click", (e) => {
  dialog.showModal(); // La méthode showModal() contient du contenu automatiquement.
});
</script>

Il y a une différence entre piéger le focus et contenir le focus. L'exemple 1 montre un piège à focus classique : lorsque l'utilisateur atteint le dernier élément focalisable et appuie sur Tab, le focus passe au premier élément. Lorsqu'il trouve le premier élément et qu'il appuie sur Shift + Tab, le focus passe au dernier élément focalisable. Il n'y a pas d'autre moyen de sortir de la boîte de dialogue que de la fermer, sauf si vous utilisez un lecteur d'écran. Nous utilisons cette technique depuis longtemps pour les fenêtres modales. C'est un moyen efficace de s'assurer que l'utilisateur ne peut pas s'échapper accidentellement de l'élément, mais il y a deux problèmes potentiels.

Tout d'abord, les utilisateurs peuvent ne pas s'attendre à ce comportement car, en général, lorsqu'ils atteignent la fin ou le début de la page, l'attention se porte sur l'interface utilisateur du navigateur.

Deuxièmement, dans l'article « Rendre les éléments focalisables », vous avez appris quels sont les éléments qui peuvent recevoir le focus. Si vous combinez cette liste avec tous les attributs qui peuvent ajouter ou supprimer cette capacité, la liste devient longue et difficile à maintenir, et certains éléments risquent d'être oubliés. La solution de l'exemple 1 ne fonctionne que parce qu'elle part du principe que la boîte de dialogue ne contient que des boutons et des liens.

Beaucoup de gens ont quand même opté pour les pièges à focus parce que, jusqu'à récemment, il n'y avait pas d'autre moyen de contenir le focus. Au début de l'année 2023, tous les principaux navigateurs ont ajouté la prise en charge de l'attribut inert. La boîte de dialogue personnalisée de l'exemple 2 ne piège pas le focus mais le contient en rendant le reste de la page inactif. Tout ce qui n'est pas la boîte de dialogue est enveloppé dans une page div avec la classe .page-wrapper. Lorsque la boîte de dialogue est ouverte, le script ajoute l'attribut inert à cette div, ce qui rend tous ses descendants inactifs. Visuellement, ils sont toujours là, mais vous ne pouvez pas les mettre en évidence et ils ne sont temporairement pas exposés dans l'arbre d'accessibilité. Cela limite l'interaction au contenu actuellement pertinent sans piéger l'utilisateur, qui peut toujours quitter la page. Cette solution est beaucoup plus proche du comportement natif et est plus sûre et plus facile à mettre en œuvre. L'élément de dialogue natif (voir exemple 3) est livré avec ce comportement par défaut.

L'attribut inert et l'élément dialogue permettent de s'assurer beaucoup plus facilement que les utilisateurs ne se perdent pas ailleurs sur la page, mais les pièges à focus peuvent encore avoir des avantages. La meilleure façon de déterminer si vous en avez besoin est de faire un test avec les utilisateurs et de décider en fonction de leurs commentaires.
