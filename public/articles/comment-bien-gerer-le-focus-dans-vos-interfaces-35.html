Comment bien gérer le focus dans vos interfaces ?

Lorsque vous déplacez l'attention des utilisateurs vers un autre endroit ou une autre couche de la page (par exemple, en affichant une boîte de dialogue modale ou une autre superposition), vous devez vous assurer qu'ils peuvent également interagir immédiatement avec ce contenu sans effort supplémentaire. Par conséquent, vous devez déplacer le focus vers cette partie de l'interface utilisateur. Si vous ne le faites pas, les utilisateurs risquent d'être encore sur un élément interactif de la page en arrière-plan. Le fait d'appuyer sur Tab aurait alors pour conséquence de focaliser des éléments qui sont obscurcis et non pertinents. Cela peut prêter à confusion et empêcher les gens d'accéder à des informations et à des fonctionnalités importantes.

Vous devez déplacer votre attention vers l'endroit où elle est actuellement nécessaire. Ce processus s'appelle la gestion de l'attention. Les exemples 1 et 2 illustrent comment créer une fenêtre modale personnalisée et déplacer le focus du bouton vers la fenêtre modale et inversement, selon les besoins.

Exemple 1 – Un bouton et une boîte de dialogue modale personnalisée

<button type="button" class="open" aria-haspopup="dialog">Se connecter</button>

<div role="dialog" aria-modal="true" hidden aria-labelledby="heading">
  <div>
    <button type="button" class="close">Fermer</button>

    <h1 id="heading" tabindex="-1">Se connecter</h1>
  </div>
</div>

<style>
/* Style de base pour le dialogue */
  [role="dialog"] {
    align-items: center;
    background: rgb( 0 0 0 / 0.2);
    inset: 0;
    justify-content: center;
    margin: auto;
    position: fixed;
  }

/* Modifie l'affichage de la boîte de dialogue lorsqu'elle est visible. */
  [role="dialog"]:not([hidden]) {
    display: flex;
  }

/* Style de base pour le contenu du dialogue */
  [role="dialog"] > div {
    background: rgb(255 255 255);
    box-sizing: border-box;
    max-width: 40rem;
    padding: 2rem;
    width: 90vw;
  }

/* Affiche le style de mise au point pour tous les éléments */
  :focus-visible {
    outline: 4px solid #123456;
    outline-offset: 0.25em;
  }

/* Déplace le contour de la boîte de dialogue à l'intérieur pour qu'il soit visible. */
  [role="dialog"]:focus-visible { 
    outline-offset: -0.5em;
  }
</style>

Exemple 2 – Exemple simple de gestion de la mise au point

const dialogOpen = document.querySelector(".open");
const dialogClose = document.querySelector(".close");
const dialog = document.querySelector('[role="dialog"]');
const heading = dialog.querySelectorAll('[tabindex="-1"]')[0];
let trigger;

dialogOpen.addEventListener("click", open);
dialogClose.addEventListener("click", close);

function open() {
    // l'élément qui a déclenché l'action.
  trigger = document.activeElement; 
    // Affiche le dialogue.
  dialogToggle(); 
}

function close() {
  // Cache la boîte de dialogue.
  dialogToggle();
}

function dialogToggle() {
  const isOpen = !dialog.hasAttribute("hidden");

  if (!isOpen) {
    dialog.removeAttribute("hidden");
        // Déplace le focus du bouton vers l'en-tête de la boîte de dialogue.
    heading.focus();
  } else {
    dialog.setAttribute("hidden", "hidden");
        // Déplace l'attention de l'en-tête vers le bouton.
    trigger.focus(); 
  }
}

Si vous utilisez la boîte de dialogue modale native en HTML, vous n'avez pas besoin de gérer le focus, comme le montre l'exemple 3.

Exemple 3 – Un bouton et une boîte de dialogue modale native

<button type="button" class="open" aria-haspopup="dialog">Se connecter</button>

<dialog aria-labelledby="heading">
  <form method="dialog">
    <button type="button">Fermer</button>
  </form>

  <h1 id="heading">Se connecter</h1>
</dialog>

<script>
const dialogOpen = document.querySelector(".open");
const dialog = document.querySelector("dialog");

dialogOpen.addEventListener("click", open);

function open() {
  dialog.showModal();
}
</script>

L'exemple 2 illustre le concept de base de la gestion du focus : vous vous souvienez du dernier élément focalisé en stockant une référence à cet élément dans une variable à l'aide de document.activeElement. Avant de déplacer le focus, vous devez décider où vous voulez le mettre. Plusieurs options s'offrent à vous :

  • Le premier élément focalisable, s'il y en a un. Cela fonctionne, mais les utilisateurs risquent d'atterrir quelque part au milieu ou à la fin du contenu, en fonction de sa position dans le DOM.
  • Le bouton de fermeture, s'il y en a un. C'est une option sûre si vous ne pouvez pas garantir qu'il y a d'autres éléments interactifs. L'inconvénient est que cela peut être irritant lorsque la première chose qu'entend un utilisateur de lecteur d'écran est « bouton de fermeture ».
  • S'il y a un titre, vous pouvez le rendre focalisable en utilisant tabindex=-1. Un titre se trouve généralement au début du contenu, et en se concentrant dessus, un lecteur d'écran annonce automatiquement son contenu textuel.
  • Comme pour l'en-tête, vous pouvez rendre le dialogue labelisé focalisable à l'aide de tabindex=-1.

Votre choix dépendra de vos possibilités et contraintes et de ce qui fonctionne le mieux pour tes utilisateurs. Je préfère me concentrer sur l'en-tête. Une fois que l'utilisateur a fini d'interagir avec le contenu de la boîte de dialogue, vous ramenez le focus sur le bouton qui l'a ouverte.

L'exemple 1 utilise une solution de dialogue personnalisée, mais en HTML, il existe également un élément de dialogue natif (voir exemple 3). Cet élément est pratique parce qu'il intègre beaucoup de choses que vous devez faire manuellement :

  • Il a un rôle implicite de dialogue.
  • Il fournit des méthodes d'ouverture et de fermeture.
  • Elle rend le reste de la page inerte.
  • Il gère le focus.

La spécification définit les règles de gestion du focus comme suit :

  • S'il existe un descendant focalisable, « focalisez-le ».
  • Si ce n'est pas le cas, concentre l'élément de dialogue lui-même.

La gestion du focus est essentielle pour rendre les interfaces utilisateur complexes accessibles.99% du temps, il ne faut pas toucher à l'ordre du focus. Vous voulez presque toujours que le focus suive l'ordre naturel et laissez les utilisateurs décider quand et où aller ensuite. Vous ne devez scénariser le focus lorsque les utilisateurs ne peuvent pas atteindre certaines parties de l'interface sans faire beaucoup d'efforts. Cela s'applique également à l'attribut autofocus en HTML, que vous pouvez également utiliser pour déplacer le focus. Vous devez certainement éviter de déplacer le focus dans la plupart des navigations et des formulaires.

Parfois, ce n'est pas l'utilisateur qui déclenche l'action. La première fois que vous visitez de nombreux sites Web européens, une boîte de dialogue s'ouvre automatiquement pour afficher une bannière de consentement aux cookies. Lorsque vous ne gèrez pas le focus dans des cas comme celui-ci, les utilisateurs peuvent interagir avec la page en arrière-plan sans pouvoir fermer la superposition .
