Exécution des actions : labelliser clairement les boutons

Un bouton peut prendre différentes formes. Le contenu enveloppé dans le bouton peut être du texte, une icône ou les deux. Lorsque vous ne donnez pas de nom à un bouton, même si la conception le prévoit, les utilisateurs de lecteurs d'écran peuvent être incapables de savoir à quoi il sert.

Tout d'abord, si le bouton contient du texte, ce texte sert d'étiquette, comme le montre l'exemple 1.

Exemple 1 – Un bouton dont le nom accessible provient de son contenu

<button type="button">Télécharger le rapport en pdf (130kio)</button>

Si le bouton contient une image , l'attribut alt de l'image doit fournir l'étiquette, comme dans l'exemple 2.

Exemple 2 – Un bouton dont le nom accessible provient de l'attribut alt de l'image

<button type="button">
    <img src="/images/download.svg" alt="Télécharger le rapport en pdf (130kio)" width="26">
</button>

Au lieu de img, vous pouvez aussi utiliser et un SVG avec un élément <title> ou <desc>. Pour la prise en charge inter-navigateurs, utilisez aria-labelledby sur le SVG et créez une référence au titre, comme le montre l'exemple 3.

Exemple 3 – Un bouton dont le nom accessible provient du SVG

<button type="button">
  <svg viewBox="0 0 39 44" aria-labelledby="title" role="img" width="26">
    <title id="title">Télécharger le rapport en pdf (130kio)</title>
    <path d="M19.5 36.5 1.6 26.1v-3.6l16.3 9.4V1.5h3.2v30.4l16.3-9.4v3.6z"/>
    <path d="M1 41.5h37" style="stroke:#000;stroke-width:3;"/>
  </svg>
</button>

Vous pouvez également retirer le graphique de l'arbre d'accessibilité en définissant un attribut alt vide sur le img ou aria-hidden="true" du SVG. Si vous faites cela, le bouton a toujours besoin d'une alternative textuelle, que vous pouvez fournir avec un texte visuellement caché (voir exemple 4), aria-labelledby, ou aria-label (voir exemple 5).

Exemple 4 – Le texte est visuellement caché

<button type="button">
  <span class="visually-hidden">Télécharger le rapport en pdf (130kio)</span>
  <img src="/images/download.svg" alt="" width="26">
</button>

<!-- ou -->

<button type="button">
  <span class="visually-hidden">Télécharger le rapport en pdf (130kio)</span>
  <svg viewBox="0 0 39 44" aria-hidden="true" width="26">
    <path d="M19.5 36.5 1.6 26.1v-3.6l16.3 9.4V1.5h3.2v30.4l16.3-9.4v3.6z"/>
    <path d="M1 41.5h37" style="stroke:#000;stroke-width:3;"/>
  </svg>
</button>

<style>
  .visually-hidden {
    clip-path: inset(50%);
    height: 1px;
    overflow: hidden;
    position: absolute;
    white-space: nowrap;
    width: 1px;
  }
</style>

Exemple 5 – L'attribut aria-label désigne le bouton

<button type="button" aria-label="Sauvegarder">
  <svg aria-hidden="true" viewBox="0 0 39 44" width="26">
    <path d="M19.5 36.5 1.6 26.1v-3.6l16.3 9.4V1.5h3.2v30.4l16.3-9.4v3.6z"/>
    <path d="M1 41.5h37" style="stroke:#000;stroke-width:3;"/>
  </svg>
</button>

La suppression des graphiques imbriqués dans l'arbre d'accessibilité est également utile lorsque vous combinez une icône avec du texte, car le texte élimine le besoin d'une étiquette supplémentaire pour l'icône, comme le montre l'exemple 6.

Exemple 6 – Combinaison de texte et d'icône

<button type="button">
  Sauvegarder
  <img src="/images/download.svg" alt="" width="26">
</button>

<button type="button">
  Sauvegarder
  <svg aria-hidden="true" viewBox="0 0 39 44" width="26">
    <path d="M19.5 36.5 1.6 26.1v-3.6l16.3 9.4V1.5h3.2v30.4l16.3-9.4v3.6z"/>
    <path d="M1 41.5h37" style="stroke:#000;stroke-width:3;"/>
  </svg>
</button>

Selon le rapport WebAIM Million 2024, qui est le résultat d'une évaluation annuelle automatisée de l'accessibilité du top 1 million de sites web, 28,2 % des sites web testés contenaient des boutons vides, ce qui en fait l'un des cinq principaux problèmes relevés par l'étude. Un bouton vide est soit un bouton sans enfant, soit un bouton qui contient des graphiques non étiquetés et sans aucune autre source que l'étiquette.

Il existe différentes méthodes pour étiqueter les boutons. Celle que vous choisissez dépend des besoins. Lorsque vous avez plusieurs options, je vous recommande de suivre cet ordre de priorité :

  1. Techniques HTML natives
  2. aria-labelledby en pointant le texte visible existant
  3. Contenu visiblement caché
  4. aria-label

Si le bouton ne contient que du texte ou une combinaison de texte et d'icônes, placez l'étiquette sous forme de texte entre la balise de début et la balise de fin de l'élément, comme le montre l'exemple 1. S'il y a une icône qui ne fournit pas d'informations supplémentaires, supprimez-la de l'arbre d'accessibilité pour éviter la redondance (voir l'exemple 6).

S'il n'y a pas de texte visible mais seulement une image ou une icône, son texte alternatif peut servir de nom au bouton (voir les exemples 2 et 3). Ce type d'images est appelé images fonctionnelles. Leur texte alternatif doit décrire non pas ce qu'elles montrent, mais leur but. Vous pouvez aussi utiliser , un texte visuellement caché(exemple 4), aria-labelledby ou l'attribut aria-label (exemple 5).

Les noms accessibles manquants sont probablement le problème le plus courant, mais les noms erronés peuvent aussi être problématiques. Parfois, ils sont dans la mauvaise langue naturelle, comme l'anglais sur un site français. Parfois, ils contiennent des variables non résolues ou du texte de remplacement. Ces problèmes surviennent généralement avec les boutons à icône seulement. Vous pouvez facilement manquer le texte ou les étiquettes visuellement cachés fournis par des attributs tels que aria-label puisqu'ils ne sont visibles que dans le code, et non dans l'interface utilisateur rendue. Vous devrez privilégier les boutons dont le texte est visible car ils sont universellement compréhensibles et moins sujets aux erreurs.

L'étiquette doit être informative et concise. N'étiquettez pas un bouton « Cliquer ici pour ouvrir ou fermer la navigation ». « Navigation » est suffisant.
