Dans la suite de ce tuto, nous allons construire, améliorer et déployer un site de blog conçu pour les développeurs. Ce projet vous initiera aux tâches les plus courantes du développement web, comme :

  • La gestion de bases de données relationnelles,
  • Le traitement et la validation des données de formulaire,
  • et bien sûr (le préféré de tous) : les tests.

À chaque chapitre, vous apprendrez une nouvelle compétence à travers des projets concrets et pratiques. Dans le tableau suivant, vous trouverez une brève description des compétences clés associées aux différentes fonctionnalités du blog.

Le cahier des charges

Il est toujours bon, lorsqu’on démarre un projet ambitieux, d’avoir une spécification fonctionnelle en tête. Pour le site de blog, notre spécification sera tout simplement la liste des fonctionnalités que nous souhaitons intégrer. Ces fonctionnalités sont basées sur mon expérience dans le développement de mon blog personnel :

  • Les articles doivent pouvoir être saisis via une interface web. Pour la mise en forme, l’auteur pourra utiliser Markdown, un langage de balisage léger et agréable visuellement.
  • Des images pourront être téléversées sur le site et facilement intégrées aux articles.
  • Les articles pourront être organisés à l’aide de tags (étiquettes) en nombre illimité.
  • Le site devra supporter plusieurs auteurs.
  • Les articles pourront être affichés par ordre de publication, mais aussi listés par mois, par tag, ou par auteur. Les longues listes d’articles seront paginées.
  • Les articles pourront être enregistrés en tant que brouillons, visibles uniquement par leur auteur jusqu’à leur publication.
  • Les visiteurs du site pourront poster des commentaires sur les articles. Ces commentaires seront vérifiés contre le spam, puis l’auteur pourra choisir de les rendre visibles ou non.
  • Des flux Atom seront disponibles pour tous les articles, y compris des flux séparés pour chaque auteur et chaque tag.
  • Les articles pourront être accédés via une API RESTful. Les auteurs disposeront d’un jeton d’API leur permettant de modifier les articles via l’API.

Bien que cette liste ne soit pas exhaustive, elle couvre les fonctionnalités principales de notre site de blog. J’espère que vous trouverez ce projet à la fois amusant et stimulant.Je suis certain que vous avez hâte de commencer, alors lançons la mise en place de notre projet de blog.

Création du projet blog

Commençons par créer un nouveau projet dans notre répertoire de travail. Sur mon ordinateur portable, il s'agit de /home/user/projects, ou, sur un système Unix, simplement ~/projects. C’est exactement ce que nous avons fait lorsque nous avons créé l’application hello_flask. Note : n'oubliez pas d'effacer l'ancien projet (hello_flask)

$ cd ~/projects
$ mkdir blog
$ cd blog

Nous allons ensuite devoir configurer notre environnement virtualenv. Cela diffère de ce que nous avons fait précédemment, car il s’agit d’une manière plus structurée d’utiliser virtualenv.

$ python -m venv .venv

L’étape suivante consiste à installer Flask dans notre environnement virtuel. Pour cela, nous allons activer le virtualenv et utiliser pip pour installer Flask.

$ source .venv/bin/activate
(.venv) ... $ pip install flask

Jusqu’à présent, tout cela devrait t’être relativement familier. Cependant, au lieu de créer un seul fichier pour notre application — ce qui est tout à fait autorisé et pertinent pour des applications très simples — nous pouvons aussi créer un nouveau dossier nommé app. Ce dossier nous permettra de structurer notre application de manière modulaire et plus logique.

À l’intérieur de ce dossier, nous allons créer cinq fichiers vides nommés : __init__.py, app.py, config.py, main.py et views.py, comme suit :

(.venv) ... $ mkdir app
(.venv) ... $ touch app/{__init__,app,config,main,views}.py

Note : La dernière commande utilise une petite astuce du shell pour créer plusieurs fichiers à partir des noms entre crochets.

Mais à quoi servent ces fichiers que nous venons de créer ? Comme vous allez le voir, chaque fichier a un rôle important. Leur nom donne déjà une idée de leur fonction, mais voici un bref aperçu des responsabilités de chaque fichier :

Structure de base d’un dossier app/ pour une application Flask
Nom du fichier Description
__init__.py Indique à Python que le dossier app/ doit être traité comme un package Python.
app.py Contient l’application Flask elle-même.
config.py Fichier destiné à stocker les variables de configuration de notre application Flask.
main.py Le point d’entrée pour exécuter l’application.
views.py Définit les routes URL et les vues associées de l’application.

Une application Flask minimaliste

Remplissons maintenant ces fichiers avec le strict minimum de code nécessaire pour obtenir une application Flask exécutable. Cela mettra notre projet en bonne position pour le chapitre suivant, dans lequel nous commencerons à coder la sauvegarde et la récupération des articles du blog depuis la base de données.

Nous allons commencer par le module config.py. Ce fichier contiendra une classe Configuration qui indiquera à Flask que nous souhaitons exécuter notre application en mode DEBUG.

Ajoutez les deux lignes de code suivantes dans le fichier config.py :

class Configuration(object):
    DEBUG = True

Ensuite, nous allons créer notre application Flask et lui indiquer d’utiliser les valeurs de configuration définies dans le module config. Ajoutez le code suivant dans le fichier app.py :

from flask import Flask
from config import Configuration


app = Flask(__name__)
app.config.from_object(Configuration)

Le module views contiendra une seule vue, liée à l’URL racine du site. Ajoutez le code suivant dans views.py :

from app import app


@app.route('/')
def homepage():
    return 'Accueil'

Comme vous l’avez probablement remarqué, il nous manque encore l’appel à app.run(). Nous placerons ce code dans main.py, que nous utiliserons comme point d’entrée de notre application. Ajoute le code suivant dans le fichier main.py :

from app import app
import views


if __name__ == '__main__':
    app.run()

Nous n’appelons pas app.run(debug=True) car nous avons déjà indiqué à Flask d’exécuter l’application en mode debug via l’objet Configuration. Vous pouvez lancer l’application depuis la ligne de commande en exécutant le module main de la façon suivante : 

(.venv) ... $ python app/main.py

Prise de recul

À l’exception de la classe Configuration, la majeure partie de ce code devrait vous sembler familière. Nous avons essentiellement repris le code de l’exemple hello_flask et l’avons réparti en plusieurs modules. Cela peut sembler un peu excessif d’écrire seulement deux ou trois lignes par fichier, mais à mesure que votre projet grandira, vous verrez à quel point cet engagement précoce en faveur d’une bonne organisation porte ses fruits. Vous avez peut-être remarqué qu’il existe une priorisation interne entre ces fichiers, basée sur l’ordre dans lequel ils sont importés — ceci afin d’éviter le risque d’importations circulaires. Une importation circulaire se produit lorsque deux modules s’importent mutuellement, ce qui empêche leur importation effective. Avec Flask, il est très facile de générer ce genre de problème, car de nombreux éléments dépendent de l’objet central app.

Pour éviter cela, certaines personnes préfèrent regrouper tout dans un seul module. Cela fonctionne pour les petites applications, mais ce n’est pas viable au-delà d’une certaine taille ou complexité. C’est pourquoi nous avons découpé notre application en plusieurs modules et créé un point d’entrée unique qui contrôle l’ordre des importations.

Le flux d’importation

L’exécution commence lorsque vous lancez la commande python main.py depuis la ligne de commande. La première ligne de code que l’interpréteur Python rencontre importe l’objet app depuis le module app. Nous nous trouvons maintenant dans app.py, qui importe Flask ainsi que notre objet Configuration. Le reste du module app.py est alors lu et interprété, puis le contrôle revient à main.py.

La seconde ligne de main.py importe le module views. Nous sommes alors dans views.py, qui dépend de app.py pour le décorateur @app.route — et cet objet est déjà disponible depuis main.py. Les routes et les vues sont enregistrées au moment où le module views est interprété, puis l’exécution revient à main.py. Puisque vous exécutez directement main.py, la condition du bloc if s’évalue à True et votre application se lance.
