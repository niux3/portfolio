Exécuter des actions : supprimer les styles de boutons par défaut

Même lorsqu'un bouton ne ressemble pas à un bouton, il doit quand même répondre à la plupart des exigences décrites dans l'article : « choisissez le bon élément ». Si ce n'est pas le cas, il risque de ne pas être accessible aux utilisateurs de claviers et de lecteurs d'écran. Si vous choisissez un élément générique avec moins de styles par défaut au lieu de l'élément <button>, les utilisateurs risquent de ne pas pouvoir accéder au faux bouton ou de ne pas comprendre son utilité.

Si vous souhaitez utiliser un bouton mais que vous ne voulez pas qu'il ressemble à un bouton, vous devez toujours utiliser l'élément <button> mais supprimer les styles de bouton par défaut. Le CSS propose trois stratégies efficaces pour supprimer les styles de bouton par défaut, comme le montrent les exemples 1, 2 et 3.

Exemple 1 – Suppression ou réinitialisation manuelle des propriétés

button {
  background: none;
  border: 0.1em solid transparent;
  font: inherit;
  padding: 0;
}

Exemple 2 – Réinitialisation de toutes les propriétés du bouton à leur valeur initiale

button {
  all: initial;
}

button:focus-visible {
  outline: 0.1em solid;
  outline-offset: 0.1em;
}

Exemple 3 – Réinitialisation de toutes les propriétés du bouton à leur valeur initiale, à l'exception des propriétés héritables.

button {
  all: unset;
}

button:focus-visible {
  outline: 0.1em solid;
  outline-offset: 0.1em;
}

Les boutons des systèmes d'exploitation et des pages Web ont une forme et un style particuliers par défaut : un rectangle avec une bordure, une couleur d'arrière-plan et un peu de remplissage entre le texte et la bordure. Lorsque vous stylisez une page, vous modifiez les valeurs par défaut et vous ajoutez éventuellement des propriétés, comme le montre l'exemple 4. Vous pouvez aussi avoir des versions de ce bouton qui varient en taille, en couleur et en forme.

Exemple 4 – Styles personnalisés pour les boutons

button {
  --color: 0.47;

  background: oklch(var(--color) 0.05 195.6);
  color: #fff;
  font-size: 1.2rem;
  font-family: inherit;
  padding: 0.4em 0.9em;
  border-radius: 3px;
  border: 0;
  min-inline-size: 7rem;
}

button:is(:hover, :focus-visible) {
  --color: 0.27;
}

Certains boutons ne ressemblent pas à des boutons car ils ne sont constitués que d'une icône. La façon dont vous implantez un tel bouton est cruciale. L'un des problèmes d'accessibilité les plus courants sur la plupart des sites Web que je vérifie concerne les faux boutons : de nombreux développeurs partent du principe que si un contrôle ne ressemble pas à un bouton, il n'a pas besoin d'être un élément <button>. Leur raisonnement est le suivant : s'il n'y a pas de style de bouton, il n'est pas nécessaire de les supprimer. Il en résulte souvent un code comme celui de l'exemple 5, qui semble inoffensif mais qui fait une différence considérable en matière d'accessibilité.

Exemple 5 – Mauvaise pratique : Un faux bouton div

<div class="button" aria-label="Navigation">
  <svg width="24" height="24" aria-hidden="true">
    <path d="M3 18h18v-2H3v2zm0-5h18v-2H3v2zm0-7v2h18V6H3z">
  </svg>
</div>

<script>
const button = document.querySelector(".button");
button.addEventListener("click", (e) => {
  console.log("active la navigation");
});
</script>

Ce « bouton » pose plusieurs problèmes :

  • Il n'est pas possible de l'utiliser au clavier.
  • Même s'il était focalisable, l'activation via Entrée ou Espace ne fonctionnerait pas.
  • Les lecteurs d'écran ne l'annoncent pas comme un bouton.
  • Certains lecteurs d'écran ne l'annoncent pas du tout.
  • Vous ne devez pas utiliser aria-label sur les éléments génériques parce qu'il n'est pas valide de les nommer.

Remarque : Dans l'exemple 5, vous avez remarqué l’utilisation de l’événement click. Le problème de cet événement, c’est qu’il ne fonctionne pas toujours correctement sur mobile ou tablette. Parfois, cela peut même entraîner un manque de performance. En effet, la souris n’existe pas sur ces types de supports.

À noter que l’événement click est déclenché uniquement après une séquence complète mousedown suivie de mouseup sur le même élément. Sur les appareils mobiles, cette séquence est simulée avec un léger délai (généralement autour de 300 ms), ce qui peut provoquer des lenteurs ou des comportements inattendus.

La solution, pour les appareils tactiles, consiste à utiliser les événements pointerdown ou pointerup. Cependant, en utilisant l’un de ces événements, sur un ordinateur de bureau ou avec un lecteur d’écran, vous perdez la possibilité d’utiliser le comportement clavier par défaut associé aux touches Enter ou Espace. Pire encore, l’événement pointerdown intervient avant que l’élément n’ait reçu le focus, ce qui empêche les styles :focus de s’appliquer correctement et nuit fortement à l’accessibilité et à la navigation clavier.

L’astuce consiste donc à détecter le type d’appareil et à appliquer l’événement le plus adapté. Dans le tableau 1, je récapitule les comportements par défaut associés à chaque interaction.

Le fait que le « bouton » de l'exemple 5 ne soit pas focalisable affecte les utilisateurs de clavier et de lecteurs d'écran, mais ces derniers peuvent utiliser le curseur virtuel pour accéder au faux bouton. Si un récepteur d'événement de clic est attaché au div, certains lecteurs d'écran indiqueront que vous ppouvez cliquer dessus. Néanmoins, si vous regardez le tableau 2, vous ppouvez constater que l'utilisation d'un div ou de tout autre élément non interactif n'est pas fiable. Si vous avez besoin de supprimer les styles des boutons par défaut, s'en tenir à l'élément <button> et réinitialiser les styles à l'aide de CSS est le choix le plus sûr.

Il ne serait pas viable d'utiliser un élément générique à la place d'un bouton ou de mettre des écouteurs d'événements sur des éléments non interactifs. Vous avez besoin d'un élément interactif si l'utilisateur peut interagir avec un composant.

Les quatre lignes de CSS que vous voyez dans l'exemple 1 sont tout ce dont vous avez besoin pour supprimer les styles d'agent utilisateur par défaut des boutons - en supposant que vous n'ayez pas ajouté d'autres règles, comme nous l'avons fait dans l'exemple 4. Dans ce cas, vous devrez également réinitialiser ces styles. C'est le problème de cette approche manuelle : vous devez suivre les changements apportés à vos styles personnalisés par défaut et ajuster les styles de réinitialisation en conséquence, ou bien réinitialiser toutes les propriétés possibles à l'avance.

L'approche illustrée dans les exemples 2 et 3 est beaucoup plus efficace. Vous ppouvez utiliser la propriété all dans CSS pour réinitialiser toutes les propriétés d'un élément, à l'exception de unicode-bidi, direction, et des propriétés personnalisées CSS. Selon vos besoins, vous pouvez définir sa valeur à initial ou unset.

Le mot-clé initial fixe toutes les valeurs des propriétés à leur valeur initiale. Chaque propriété a une valeur initiale, définie dans la table de définition de la propriété. Par exemple, si vous regardez la propriété color dans la spécification, la valeur initiale définie dans la table de définition est CanvasText. Notez que la valeur initiale n'est pas la valeur par défaut de la propriété définie dans l'agent utilisateur.

Le mot-clé unset réinitialise une propriété à sa valeur héritée, si la propriété hérite naturellement de son parent, et à sa valeur initiale dans le cas contraire. Cela peut être utile si vous voulez conserver des propriétés spécifiques comme font-family ou color.

Quoi qu'il en soit, vous devrez peut-être rétablir les styles focus et hover, car all les réinitialise également. L'essentiel est que même si un bouton ne ressemble pas à un bouton, il doit se comporter comme tel.
