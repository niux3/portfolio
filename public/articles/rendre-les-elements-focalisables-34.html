Rendre les éléments focalisables

Les personnes qui dépendent de l'accessibilité au clavier naviguent en utilisant la touche Tab, ou un dispositif physique qui correspond à la touche, pour passer d'un élément interactif à un autre. Lorsqu'elles utilisent un site Web, elles doivent pouvoir effectuer les mêmes actions que les utilisateurs de souris. Si vous ne prenez pas de mesures spécifiques pour vous en assurer, ils risquent de ne pas pouvoir atteindre certaines parties de l'interface. Vous pouvez rendre les éléments non focalisables focalisables si nécessaire, comme le montre l'exemple 1.

Exemple 1 – Rendre un panneau à onglets focalisable

<div class="tabs">
  <h3 id="tablist-1">Résultat élections</h3>
  <div role="tablist" aria-labelledby="tablist-1">
    <button id="tab-1" type="button" role="tab" aria-selected="true" aria-controls="tabpanel-1">
      Graphique
    </button>
    <button id="tab-2" type="button" role="tab" aria-selected="false" aria-controls="tabpanel-2" tabindex="-1">
      Tableau
    </button>
  </div>

  <div id="tabpanel-1" role="tabpanel" tabindex="0" aria-labelledby="tab-1">
    <!-- graphique -->
  </div>
  <div id="tabpanel-2" role="tabpanel" tabindex="0" aria-labelledby="tab-2" class="is-hidden">
    <!-- tableau -->
  </div>
</div>

Vous pouvez éviter de nombreux problèmes d'accessibilité au clavier avant qu'ils ne se posent en utilisant du HTML sémantique. Plusieurs éléments sont destinés à un contenu interactif et sont accessibles au clavier par défaut (voir tableau 1).

Les deux solutions de l'exemple 2 sont sémantiquement les mêmes et peuvent paraître identiques avec le CSS adéquat. La grande différence est que seul le bouton natif est accessible via le clavier. Même si le bouton personnalisé était focalisable, vous ne pourrez pas effectuer d'action sur lui parce qu'il n'a pas d'événements de touche. J'en ai déjà parlé dans ces articles :

  • Exécuter des actions : éviter de désactiver les boutons

Exemple 2 – Un bouton natif et un bouton personnalisé

<button> 
  Ouvrir
</button>

<div role="button"> 
  Ouvrir
</div>

<style>
  button,
  [role="button"] {
    background: #123456;
    color: #FFFFFF;
    border: none;
    display: inline-block;
    padding: 0.5em 1rem;
    font-family: inherit;
    font-size: 1.1rem;
    line-height: 1;
  }

  :focus-visible {
    outline: 4px solid #123456;
    outline-offset: 4px;
  }
</style>

L'utilisation du HTML natif est la meilleure approche. Vous obtienez généralement une excellente accessibilité de base, mais vous devez parfois rendre focalisables des éléments qui ne le sont pas par défaut. Par exemple :

  • Lorsque vous devez déplacer le focus, mais que l'élément cible n'est pas focalisable ou n'a pas de descendants focalisables.
  • Quand vous voulez rendre les zones défilables accessibles au clavier
  • Lorsque vous construisez un élément interactif personnalisé et que vous avez de très bonnes raisons de ne pas utiliser un élément sémantique.

L'attribut tabindex

Le code de l'exemple 1 montre une solution JavaScript personnalisée pour un composant d'onglets. Essayez la démo sur le site Web ARIA Authoring Practices Guide (APG) du W3C, d'où provient ce code. Vous apprendrez que vous pouvez interagir avec les boutons de la liste des onglets en utilisant les touches fléchées gauche et droite. Pour accéder au contenu d'un onglet, vous pouvez utiliser la touche Tab, et le panneau de l'onglet correspondant recevra alors le focus. Le panneau à onglets est un élément <div> avec un rôle personnalisé tabpanel. Il peut être mis en évidence uniquement parce que la page utilise l'attribut tabindex avec une valeur 0. Cet attribut et cette valeur rendent un élément focalisable et l'ajoutent à l'ordre naturel des onglets.

Vous remarquerez également que les boutons ont un attribut tabindex avec la valeur -1, ce qui peut être utile dans deux cas :

  • lorsque vous souhaitez rendre un élément générique focalisable à l'aide de la méthode focus() de JavaScript, sans le rendre accessible au clavier ;
  • ou lorsque vous souhaitez retirer un élément de l'ordre de tabulation tout en étant capable de lui donner le focus via JavaScript.

Les boutons sont focalisables par défaut, mais dans cet exemple, appuyer sur la touche Tab doit permettre d'ignorer les boutons et de focaliser à la place le panneau de tabulation actif.
Le réglage tabindex=-1 rend cela possible, tout en vous permettant de mettre l'accent sur les boutons lorsque l'utilisateur appuie sur les touches fléchées.

L'utilisation de tabindex avec les valeurs 0 ou -1 est une pratique courante et ne pose pas de problème, à condition de l'appliquer à un élément sémantique et labellisé. Les problèmes surviennent lorsqu'un nombre positif supérieur à 0 est utilisé, comme le montre l'exemple 3. Un utilisateur de clavier navigue sur un site Web en sautant d'un élément interactif à l'autre dans un ordre séquentiel. La touche Tab suit toujours l'ordre des éléments dans le document, sauf si vous utilisez tabindex avec une valeur positive.

Exemple 3 – Mauvaise pratique : Boutons avec un ordre de tabulation personnalisé

<button tabindex="2">Bouton 1</button>
<button tabindex="1">Bouton 2</button>
<button tabindex="3">Bouton 3</button>

Les boutons de l'exemple 3 reçoivent le focus dans l'ordre suivant : Bouton 2, Bouton 1, Bouton 3.
L'ordre personnalisé défini par les attributs tabindex remplace l'ordre du DOM. Si seuls certains éléments possèdent l'attribut, le focus atteindra d'abord ces éléments, en les parcourant selon leur valeur de tabindex, puis il parcourra les éléments sans l'attribut dans l'ordre séquentiel.Dans l'exemple 4, cela donnerait : Bouton 2, Bouton 4, Bouton 1, Bouton 3, Bouton 5. Le maintien d'un ordre significatif devient complexe dès que l'on introduit des attributs tabindex avec une valeur supérieure à zéro.

Exemple 4 – Mauvaise pratique : Seuls deux boutons ont un attribut positif personnalisé

<button>Bouton 1</button>
<button tabindex="1">Bouton 2</button>
<button>Bouton 3</button>
<button tabindex="2">Bouton 4</button>
<button>Bouton 5</button>

L'attribut tabindex est puissant, polyvalent et utile pour améliorer l'accessibilité au clavier, mais il peut également s’avérer dangereux. L'utilisation de valeurs positives devient rapidement incontrôlable. Suivez les règles suivantes :

  • Utilisez tabindex="-1" pour rendre les éléments non focalisables, focalisables via JavaScript uniquement, ou pour rendre des éléments focalisables non focalisables.
  • Utilisez tabindex="0" pour rendre les éléments non focalisables accessibles au clavier.
  • Évitez d'utiliser des valeurs positives.

Éléments focalisables

Savoir quels éléments sont focalisables peut être utile, mais les énumérer tous est plus difficile qu'il n'y paraît. En plus de la liste des éléments interactifs mentionnée au début de cet article, il faut également prendre en compte les éléments ayant un tabindex positif. Il y a aussi l'attribut contenteditable, qui rend les éléments modifiables. Tout cela n'est valable que si les éléments ne sont pas cachés, désactivés ou inertes, et s'ils ne possèdent pas un tabindex avec une valeur négative.
