Il est parfois souhaitable d'augmenter la zone de clic et de toucher d'un lien pour qu'elle s'étende sur plusieurs éléments. Cela peut améliorer l'expérience utilisateur pour les utilisateurs de la souris et du toucher, mais peut avoir un effet négatif sur l'accessibilité et l'UX pour les utilisateurs du clavier et du lecteur d'écran. En fonction de la solution retenue, les problèmes suivants peuvent survenir :

  • Le texte d'un lien peut être long, ce qui rend l'utilisation d'une interface fastidieuse.
  • Les liens redondants polluent l'interface et rendent l'orientation et la navigation plus complexes pour les utilisateurs de lecteurs d'écran.
  • Des paquets de tabulation supplémentaires rendent l'utilisation d'une interface plus exigeante physiquement pour les utilisateurs de clavier
  • Les liens vides ou verbeux peuvent être difficiles ou impossibles à sélectionner pour les utilisateurs vocaux..
  • Les paramètres par défaut du navigateur, comme la sélection du texte, le clic du milieu pour ouvrir un lien dans une nouvelle fenêtre/un nouvel onglet, l'ouverture du menu contextuel ou la prévisualisation des URL, peuvent ne pas fonctionner.

Malheureusement, il n'existe pas de méthode parfaite pour relier des groupes d'éléments ; les solutions sont toutes assorties de compromis. Cet article énumère cinq techniques de relier plusieurs éléments, en commençant par la plus simple. Si vous ne voulez pas les passer toutes en revue, passe à la technique du pseudo élément, que je recommande pour la plupart des cas d'utilisation.

Pour illustrer le problème, j'ai créé un composant de carte. c'est une accroche de page généralement composée d'un titre, d'une image et d'un texte, que je lie différemment dans chaque exemple. Le code de l'exemple 1 est le code de base. Les exigences pour la carte sont :

  • Toute la carte doit être cliquable.
  • L'annonce dans les lecteurs d'écran doit être succincte.
  • La carte doit contenir le moins de taquets de tabulation possible.
  • La solution doit prendre en charge plusieurs liens différents.
  • Le code HTML doit être valide.
  • Le menu contextuel et les autres paramètres par défaut du navigateur devraient fonctionner comme prévu.
  • Le texte doit pouvoir être sélectionné.

Les avantages

  • Toute la carte est cliquable.
  • Les paramètres par défaut du navigateur, comme le clic droit ou le clic du milieu, fonctionnent comme prévu.

L'utilisation de tabindex="-1" et role="presentation" sur les liens redondants semble être une solution intelligente, mais il y a des mises en garde avec cette technique.

Les deux inconvénients importants de cette technique sont qu'elle crée de nombreux liens redondants avec la même cible et qu'elle augmente le nombre d'arrêts de tabulation.

Vous pouvez mettre tabindex=-1 sur les liens supplémentaires pour réduire le nombre d'arrêts de tabulation, comme illustré dans l'exemple 5, mais cela enfreint la troisième règle d'ARIA. Tous les contrôles des éléments interactifs doivent être utilisables avec le clavier. Vous pouvez enfreindre la deuxième règle d'ARIA et changer le rôle du lien en none ou presentation. En théorie, vous obtiendrez un élément textuel générique non focalisable qui se comporterait comme un lien pour les utilisateurs de la souris, du toucher et de la voix. En pratique, cela ne fonctionne pas, car les API d'accessibilité ignorent role=presentation et role=none sur les liens hypertextes. Il en résulte un méli-mélo confus où les liens ne peuvent pas être découverts via Tab, mais le sont avec le curseur virtuel.

Les avantages

  • Il n'y a pas d'annonce verbale lorsque vous utilisez la touche Tab avec un lecteur d'écran.
  • Certaines parties de la carte sont cliquables.
  • Les paramètres par défaut du navigateur, comme le clic droit ou le clic du milieu, fonctionnent comme prévu.
  • La carte prend en charge plusieurs liens.
  • Il est accessible par la voix.
  • Le texte peut être sélectionné.

Les inconvénients

  • Seules certaines parties de la carte sont cliquables.
  • La technique ajoute des tabulations supplémentaires.
  • La carte contient de nombreux liens redondants avec la même cible.

Pour rendre des éléments spécifiques sélectionnables, vous pouvez utiliser z-index pour les placer au-dessus du pseudo-élément. N'oubliez pas qu'ils ne seront plus liés lorsque vous ferz cela, comme le montre l'exemple 8.

Exemple 8 – Déplacement des éléments sélectionnés vers le haut d'un calque

/* etc. */

Cette technique est ma préférée pour la plupart des cas d'utilisation car elle ne nécessite pas de balisage supplémentaire et elle répond à la plupart des exigences de la liste. Le seul problème est que le pseudo-élément recouvre tous les autres éléments, ce qui signifie que le texte n'est pas sélectionnable. Vous pouvez déplacer les éléments au-dessus de la pseudocouche en appliquant un plus élevé z-index, ce qui les rend sélectionnables (l'exemple 8).

Les inconvénients

  • Le texte n'est pas sélectionnable, sauf si vous utilisez un z-index sur les autres éléménts.
  • Si des parties de la carte utilisnet un z-index avec des éléments interactifs comme des liens, il est possible que l'utilisateur se trompe et clique sur la carte au lieu du lien.

Javascript

Le balisage est le même que dans le composant de base de l'exemple 1 et vous utilisez JavaScript pour rendre toute la carte cliquable. J'ai choisi deux solutions différentes pour vous montrer comment vous pouvez y parvenir.

Dans cet exemple 9, vous récupérez la date sur mousedown et mouseup. Au lever de la souris, vous soustrayez le premier temps du second. Si le résultat est inférieur à 200 ms, il est probable que l'utilisateur n'a pas sélectionné le texte mais qu'il a simplement cliqué dessus ; et s'il est supérieur, il est probablement en train de le sélectionner.

Exemple 9 – Utilisation d'un seuil de temporisation pour détecter si un texte a été cliqué ou sélectionné.

const card = document.querySelector(".card");
let down,
    up,
    link = card.querySelector(".readmore");

card.addEventListener('mousedown', e => down = +new Date());
card.addEventListener('mouseup', e =>{
    up = +new Date();
    if (up - down < 200) {
        link.click();
    }
});

Vous avez la possibilité de détecter si le texte a été sélectionné via window.getSelection. Dans l'exemple 10, je vous montre comment y parvenir.

Exemple 10 – Utilisation de l'API de sélection pour détecter si un texte a été cliqué ou sélectionné

const card = document.querySelector(".card");
const link = card.querySelector(".readmore");

card.addEventListener("click", (e) => {
  const noTextSelected = !window.getSelection().toString();
  if (noTextSelected) {
    link.click();
  }
});

Le plus gros inconvénient de cette technique est que les comportements par défaut des navigateurs, comme le clic droit ou le clic moyen ou les aperçus d'URL au survol, ne fonctionnent plus.

Les avantages

  • Il n'y a pas d'annonce verbale lorsque vous utilisez la touche Tab avec un lecteur d'écran.
  • Toute la carte peut être cliquable.
  • Il n'y a que deux tabulation.
  • La solution prend en charge plusieurs liens différents.
  • Le HTML serait valide même s'il y avait des liens supplémentaires.
  • C'est accessible par la voix.
  • Les autres éléments de la carte sont accessibles.
  • Vous n'avez pas besoin d'un élément HTML supplémentaire.
  • Le texte peut être sélectionné.

Les inconvénients

  • Le menu contextuel et les autres options par défaut du navigateur ne fonctionnent que sur les liens directement.
  • Il s'agit d'une technique JavaScript magique. L'utilisation de bidouilles en JavaScript est astucieuses pour contourner le comportement par défaut du navigateur sujette à des erreurs.
